# 1. قواعد عمل الموظفين

## 1.1 قبل أي عمل

| # | القاعدة |
|---|---------|
| E1 | لا يبدأ أي موظف العمل قبل قراءة `PROJECT_HANDBOOK.md` بالكامل |
| E2 | كل مهمة لها **مالك واحد** — المسؤولية لا تُقسّم ضبابياً |
| E3 | إن لم تفهم المهمة — **اسأل قبل التنفيذ** |
| E4 | **التوثيق جزء من العمل** — ليس خطوة اختيارية بعد الانتهاء |

## 1.2 دورة العمل الرسمية

جميع الموظفين يلتزمون بالتسلسل التالي — **لا تجاوز لمرحلة قبل اكتمال السابقة** إلا بقرار رسمي:

```
Idea → Architecture → Documentation → Design → Database
  → Development → Testing → Content → Deployment → Maintenance
```

## 1.3 مستويات الصلاحية

| المستوى | من يملكه | نطاق العمل |
|---------|----------|------------|
| **L1 — تنفيذ** | موظف جديد، متعاون، AI Agent | مهام محددة النطاق ضمن ملفات مفتوحة |
| **L2 — تطوير** | مطور معتمد | بناء ميزات ضمن المرحلة الحالية |
| **L3 — منتج** | قائد المنتج | أولويات، قبول ميزات، فتح مناطق محمية |
| **L4 — استراتيجي** | المؤسس | قرارات المراحل، الفلسفة، توسع السوق |

## 1.4 مبادئ التقييم

- **الجودة** تُقاس بـ: هل الميزة تعمل؟ هل موثّقة؟ هل لا تكسر الفانل؟
- **الالتزام بالمرحلة** أهم من «إنجاز سريع»
- **Placeholder** مقبول مؤقتاً — **ادعاء الاكتمال** غير مقبول
- **الأخطاء** تُعالج: توثيق → إصلاح → تحديث الوثائق إن لزم

## 1.5 العمل مع أدوات AI (Lovable / Cursor)

| القاعدة | التطبيق |
|---------|---------|
| AI ينفّذ | ضمن قواعد الدستور — لا يقرّر استراتيجياً |
| Git | لا force-push أو تعديل history المدفوع |
| Commit | فقط عند طلب صريح من المؤسس أو قائد المنتج |
| مناطق محمية | أي تغيير في Landing / Quiz / Admin يتطلب موافقة L3+ |

# 2. حدود مسؤولية كل موظف

## 2.1 الفريق

المشروع يعمل بواسطة **13 موظفاً** متخصصاً. لكل موظف صلاحيات ومسؤوليات وحدود ودليل مستقل — داخل `docs/employees/`.

> **هذه الوثيقة عامة.** التفاصيل التخصصية في الدليل الفردي لكل موظف.

## 2.2 حدود الأدوار (ملخص تشغيلي)

| الدور | داخل مسؤوليته | خارج مسؤوليته |
|-------|---------------|---------------|
| **المؤسس / الكوتش** | الرؤية، المحتوى التدريبي، تجربة العضو، القرار الاستراتيجي النهائي | تعديل كود مباشر دون مراجعة |
| **قائد المنتج (Product Owner)** | أولويات المراحل، قبول الميزات، تنسيق الفرق | تغيير Landing / Quiz / Admin دون موافقة المؤسس |
| **Architect** | قرارات معمارية — ضمن دورة المشروع | قرارات منتج أو تصميم |
| **المطور (Developer)** | بناء المنصة، DB، Edge Functions، الجودة التقنية | تغيير استراتيجية المنتج أو الفانل |
| **مصمم (Designer)** | هوية بصرية، UX موبايل، Design System | إطلاق هوية تتعارض مع القرارات المعتمدة |
| **منتج المحتوى** | تمارين، تغذية، فيديوهات، نصوص عربية | نشر محتوى دون مراجعة الكوتش |
| **الأدمن التشغيلي** | مراجعة مدفوعات، دعوات، دعم عملاء | قبول / رفض دون الإجراء المعتمد |
| **موظف جديد / AI Agent** | مهام محددة النطاق، توثيق | قرار معماري أو تغيير في مناطق محمية |

## 2.3 مناطق الملكية

| المنطقة | المالك | حالة التعديل |
|---------|--------|--------------|
| Landing `/` | المؤسس + تسويق | 🔒 محمية — تعديل بطلب صريح |
| Quiz `/quiz` | المؤسس + منتج | 🔒 محمية — تعديل بطلب صريح |
| Admin `/admin/*` | عمليات | 🔒 محمية — تعديل بطلب صريح |
| المنصة `/app/*` | تطوير + منتج | ✅ مفتوحة ضمن المرحلة الحالية |
| مكتبة التمارين `scripts/` | محتوى + تطوير | ✅ مفتوحة |
| الوثائق `docs/` | الجميع | ✅ تحديث إلزامي عند تغيير جوهري |

## 2.4 القاعدة الذهبية — Golden Rule

**أي موظف يعمل فقط داخل حدود اختصاصه.**

لا يحاول اتخاذ قرار أو التفكير بدلاً من موظف آخر إذا كان الطلب خارج مسؤوليته.

# 3. تحويل المهام بين الموظفين

## 3.1 قبل البدء — checklist إلزامي

كل مهمة جديدة يجب أن تحتوي:

- [ ] **الهدف** — ماذا نريد تحقيقه؟
- [ ] **النطاق** — أي ملفات / مسارات؟ أي مناطق محمية؟
- [ ] **المرجع** — أي فصل في Master Documentation؟
- [ ] **معايير القبول** — كيف نعرف أنها اكتملت؟
- [ ] **المالك** — من المسؤول؟
- [ ] **الموعد** — متى؟

## 3.2 عند التسليم

| العنصر | مطلوب |
|--------|-------|
| ملخص التغيير | ماذا تغيّر ولماذا |
| حالة البناء | `npm run build` ناجح — للمهام التقنية |
| تحديث الوثائق | إن تغيّرت حقيقة → Master Documentation |
| المخاطر المعروفة | ما لم يُختبر أو يحتاج متابعة |
| الخطوة التالية | ماذا يأتي بعد هذا العمل |

## 3.3 بروتوكول التسليم بين موظفين

**المالك السابق يُسلّم:**

1. سياق القرار (لماذا بدأنا)
2. ما اكتمل وما لم يكتمل
3. الملفات / المناطق المتأثرة
4. Blockers مفتوحة
5. مرجع Master Documentation المحدّث

**المالك الجديد يُقرّ بـ:**

1. قراءة الدستور (إن لم يقرأ)
2. قراءة فصول Master Doc ذات الصلة
3. فهم معايير القبول
4. تأكيد النطاق — **لا توسيع صامت**

## 3.4 ممنوع في التسليم

- «اتصلت به» دون توثيق
- تسليم بدون معرفة حالة البناء (للمهام التقنية)
- تسليم مع تغييرات في مناطق محمية دون إفصاح
- افتراض أن الطرف التالي يعرف السياق

# 4. اتخاذ القرار

## 4.1 هرم القرارات

| نوع القرار | من يملكه |
|------------|----------|
| **استراتيجي** — مرحلة، فلسفة، تسعير | المؤسس فقط |
| **منتج** — ميزة، أولوية، UX جوهري | قائد المنتج + استشارة المؤسس |
| **تقني** — بنية، migration، مكتبة | المهندس + موافقة قائد المنتج |
| **تشغيلي** — دفع، دعم عميل، إيميل | الأدمن التشغيلي ضمن الإجراء المعتمد |

## 4.2 قواعد ملزمة

| # | القاعدة |
|---|---------|
| D1 | `MASTER_PROJECT_DOCUMENTATION.md` = مصدر الحقيقة التقنية — لا افتراضات من الذاكرة |
| D2 | `PROJECT_HANDBOOK.md` = مصدر الحقيقة الإدارية — لا استثناءات صامتة |
| D3 | عند الشك: **اسأل قبل البناء** — خاصة في المناطق المحمية |
| D4 | لا قرار يغيّر السلوك دون توثيق في الوثيقة المناسبة |
| D5 | **المرحلة الحالية تحدد النطاق** — لا Phase 4 قبل إكمال Phase 2 |
| D6 | **القرارات الثابتة** لا تُعاد مناقشتها دون سبب استراتيجي |

## 4.3 تصعيد القرار

```
1. الموظف يوثّق السؤال / الخيارات (2–3 بدائل كحد أقصى)
2. يرفع لمالك المنطقة
3. إن تعارضت المناطق → قائد المنتج
4. إن تعارضت الاستراتيجية → المؤسس
```

## 4.4 من يقرر ماذا — Golden Rule

| نوع القرار | من يقرر |
|------------|---------|
| معماري | Architect فقط |
| تصميم | Designer فقط |
| منتج | Product Owner فقط |
| كود | Developer فقط |

| # | القاعدة |
|---|---------|

| C2 | **لا وعود بميزة غير مبنية** — راجع Master Documentation للحالة الفعلية |
| C3 | **تسليم المهام بنطاق واضح**: ماذا، لماذا، أين، متى، من |

```

↓
PROJECT_TASKS/               ← مهام حالية
↓
Codebase                     ← تنفيذ — لا يغيّر القرارات
```

# 6. قواعد الردود

## 6.1 رد التحويل — خارج المسؤولية

عند استلام طلب خارج الاختصاص، **لا ترفض بصمت ولا تنفّذ**:

```
هذه المهمة ليست ضمن مسؤوليتي.

الموظف المناسب:
[دور الموظف — من §2.2]

سبب التحويل:
[سبب واضح ومحدد]
```

## 6.2 رد طلب توضيح — قبل التنفيذ

```
أحتاج توضيحاً قبل البدء:

1. [سؤال محدد]
2. [سؤال محدد]

المرجع المطلوب: [PROJECT_HANDBOOK / MASTER_DOC — فصل X]
```

## 6.3 رد تصعيد قرار

```
هذا القرار يتجاوز صلاحيتي (L[X]).

نوع القرار: [استراتيجي / منتج / تقني / تشغيلي]
الخيارات المقترحة:
  أ) ...
  ب) ...
  ج) ...

أرفعه إلى: [مالك المنطقة / قائد المنتج / المؤسس]
```

## 6.4 رد رفض تنفيذ

```
لا أستطيع تنفيذ هذا الطلب للأسباب التالية:

1. [سبب — انظر §8]
2. [سبب]

البديل المقترح: [إن وُجد]
```

## 6.5 قواعد عامة للردود

| # | القاعدة |
|---|---------|
| R1 | الرد **مكتوب** — في الوثيقة أو TASK أو Git — ليس شفهياً فقط |
| R2 | الرد **محدد** — لا «سأرى» أو «ربما» |
| R3 | الرد **يحيل** — دور، وثيقة، أو مرحلة — لا حكم شخصي |
| R4 | لا وعود للعضو أو للفريق بميزة غير مبنية |

# 7. ما يُمنع على أي موظف

## 7.1 ممنوع على الجميع

| # | الممنوع |
|---|---------|
| P1 | البدء بالعمل دون قراءة `PROJECT_HANDBOOK.md` |
| P2 | تنفيذ مهمة بدون مالك واحد واضح |
| P3 | تجاوز مرحلة في دورة المشروع قبل اكتمال السابقة — دون قرار رسمي |
| P4 | قرارات شفهية دون تسجيل في الوثيقة المناسبة |
| P5 | ادعاء اكتمال ميزة Placeholder أو غير مبنية |
| P6 | تعديل تخصص موظف آخر |
| P7 | force-push أو تعديل Git history المدفوع |
| P8 | Commit دون طلب صريح من المؤسس أو قائد المنتج |

## 7.2 ممنوع حسب الدور — Golden Rule

| الدور | الممنوع عليه |
|-------|--------------|
| غير Architect | اتخاذ قرار معماري |
| غير Designer | اتخاذ قرار تصميم |
| غير Product Owner | اتخاذ قرار منتج |
| غير Developer | تنفيذ كود |
| L1 | قرار معماري، تغيير مناطق محمية، commit |
| AI Agent | قرار استراتيجي، تغيير Landing / Quiz / Admin دون L3+ |

## 7.3 ممنوع في المناطق المحمية

| المنطقة | الممنوع |

| Admin `/admin/*` | أي تعديل دون طلب صريح |

## 7.4 ممنوع في التسليم

- «اتصلت به» دون توثيق
- تسليم بدون معرفة حالة البناء

# 8. متى يرفض الموظف تنفيذ الطلب

## 8.1 أسباب رفض إلزامية

| # | الحالة | الإجراء |
|---|--------|---------|
| 1 | الطلب **خارج اختصاصه** | رفض + تحويل (§9) — لا تنفيذ |
| 2 | الطلب **يتجاوز مستوى صلاحيته** (L1–L4) | رفض + تصعيد (§4.3) |
| 3 | الطلب **يخالف قراراً ثابتاً** | رفض + إحالة لقائد المنتج أو المؤسس |
| 4 | الطلب **يتجاوز مرحلة المشروع** (مثلاً Phase 4 قبل Phase 2) | رفض + توثيق السبب |
| 5 | الطلب **يستهدف منطقة محمية** دون موافقة | رفض حتى الموافقة L3+ |
| 6 | الطلب **بدون معايير قبول أو مالك** | رفض + طلب checklist (§3.1) |
| 7 | الطلب **يقتضي ادعاء اكتمال** لمحتوى Placeholder | رفض — Placeholder يُعلَن صراحة |
| 8 | الطلب **يخالف Golden Rule** | رفض — لا قرار بديل عن الدور المختص |

## 8.2 ما ليس رفضاً

| الحالة | الإجراء الصحيح |
|--------|----------------|
| لا أفهم المهمة | **طلب توضيح** — §6.2 — ليس رفضاً |
| أحتاج موافقة | **تصعيد** — §4.3 — ليس رفضاً |
| المهمة ليست لي | **تحويل** — §9 — ليس رفضاً |

# 9. متى يُحوّل المهمة إلى موظف آخر

## 9.1 يُحوّل فوراً عندما

| # | الحالة | يُحوَّل إلى |
|---|--------|-------------|
| 1 | قرار **استراتيجي** | المؤسس |
| 2 | قرار **منتج** — ميزة، أولوية، UX | Product Owner |
| 3 | قرار **معماري** | Architect |
| 4 | قرار **تصميم** — UI، هوية، tokens | Designer |
| 5 | **تنفيذ كود** — build، migration، Edge Function | Developer |
| 6 | **محتوى تدريبي** — تمارين، تغذية، فيديو | منتج المحتوى (+ مراجعة الكوتش) |
| 7 | **مدفوعات / دعم عميل / إيميل** | الأدمن التشغيلي |
| 8 | **منطقة محمية** — Landing / Quiz / Admin | مالك المنطقة (§2.3) |
| 9 | **تعارض بين منطقتين** | قائد المنتج |
| 10 | **تعارض استراتيجي** | المؤسس |

## 9.2 آلية التحويل

```
1. استخدم قالب الرد — §6.1
2. حدّد الدور المناسب — §2.2
3. اذكر سبب التحويل — محدد، لا عام
4. أرفق السياق: checklist §3.1 إن وُجد
5. لا تبدأ العمل — حتى يؤكد المالك الجديد
```

## 9.3 لا تُحوَّل — بل تُنفَّذ

| الحالة | السبب |
|--------|-------|
| المهمة ضمن اختصاصك وصلاحيتك | أنت المالك |
| المهمة في منطقة مفتوحة ضمن مرحلتك | نطاق عملك |
| طلب توضيح فقط | ليست تحويلاً |

# 10. معايير الجودة الملزمة

## 10.1 جودة العمل — جميع الموظفين

| المعيار | الحد الأدنى |
|---------|-------------|
| **الوضوح** | كل مهمة لها مالك، هدف، معايير قبول |
| **الصدق** | Placeholder يُعلَن Placeholder — لا ادعاء اكتمال |
| **التوثيق** | قرار مكتوب في الوثيقة الصحيحة |
| **الالتزام بالمرحلة** | Phase 2 = الأولوية — لا تشتيت |

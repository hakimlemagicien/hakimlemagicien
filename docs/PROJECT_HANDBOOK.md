# 1. الرؤية — Vision

**أن نكون المنصة العربية الرائدة للتدريب والتغذية الرقمية المخصصة** — حيث يحصل كل عضو على تحليل شخصي، برنامج يومي، ومتابعة حقيقية داخل منتج واحد متكامل.

نبني ليس موقعاً تسويقياً ولا ملف PDF، بل **منصة يومية عربية أصيلة** تجمع: التحليل الشخصي، الاشتراك، البرنامج، التغذية، التقدم، والدعم — تحت إشراف كوتش حقيقي.

> للتفاصيل التقنية عن حالة المنتج الحالية: Master Documentation — الفصل 1 و 2.

---

# 2. الرسالة — Mission

**تحويل كل زائر عربي إلى عضو يحقق أهدافه الجسدية عبر رحلة رقمية واضحة وموثوقة:**

```
تحليل شخصي مجاني → اشتراك مدفوع → برنامج مخصص → متابعة يومية داخل المنصة
```

نخدم رجالاً ونساءً في الخليج والمغرب العربي وما بعدهما — بمحتوى عربي RTL أصيل، وليس ترجمة. المنتج **رقمي بالكامل**؛ ليس تدريباً حضورياً.

# 3. القيم الجوهرية — Core Values

| القيمة | المعنى العملي |
|--------|---------------|
| **الأصالة العربية** | كل نص، صورة، وتجربة مبنية للعرب RTL — لغة، ثقافة، وسياق محلي |
| **الصدق مع العضو** | لا نعرض ميزة كمكتملة إن كانت Placeholder؛ نميّز بوضوح بين الموجود والمخطط |
| **الكوتش أولاً** | Hakim ليس شعاراً — هو مصدر الثقة: صور تحوّل، دعم WhatsApp، ومتابعة حقيقية |
| **الجودة قبل السرعة** | نُطلق مراحل مكتملة (Phase by Phase) بدل ملء المنصة بمحتوى وهمي |
| **الأمان والخصوصية** | بيانات العملاء والإيصالات محمية؛ لا اختصارات في RLS أو الوصول |
| **الوضوح التشغيلي** | كل قرار موثّق؛ كل مهمة لها مالك؛ لا عمل ضبابي |
| **الاستدامة التقنية** | نبني على أساس حديث قابل للتوسع — لا حلول مؤقتة تتحول لديون دائمة |

# 4. فلسفة المنتج — Product Philosophy

## 4.1 المبدأ المركزي

المنتج ليس «برنامج تمرين» منفصل — هو **نظام تحويل كامل**: من أول زيارة للموقع حتى الالتزام اليومي داخل `/app`.

## 4.2 طبقات المنتج

| الطبقة | الدور | المبدأ |
|--------|-------|--------|
| **التسويق** | Landing + Quiz | تحليل شخصي عميق يبني الثقة قبل الدفع |
| **التحويل** | Checkout + Admin | دفع بنكي يدوي مناسب للسوق العربي — شفافية كاملة |
| **العضوية** | Free vs Premium | محتوى مجاني يجذب؛ Premium يُفتح بقيمة حقيقية لا بقفل فقط |
| **المنصة اليومية** | `/app` | التمرين، التغذية، التقدم، الدعم — في مكان واحد |
| **المحتوى** | تمارين + تغذية | مكتبة منظمة قابلة للتوسع — ليست نصوصاً عشوائية |

## 4.3 ما لا نفعله

- لا نبيع وهم التخصيص — إن لم يكن البرنامج مبنياً، لا ندّعي العكس
- لا نوسّع المنتج أفقياً (مدونة، متجر، تطبيق) قبل إكمال **العمود الفقري**: برنامج + تغذية + عضوية موثوقة
- لا نفعّل طرق دفع إلكترونية في الواجهة قبل اكتمال التدفق الحالي
- لا نحوّل المنصة لتطبيق حضوري أو خدمات في الموقع

## 4.4 تمييز المنتج

- عربي أصيل RTL — ليس ترجمة
- فانل كويز عميق مع تخصيص بصري ذكر/أنثى
- دفع بنكي (الإمارات + المغرب) — مناسب للسوق
- كوتش حقيقي مع تحوّلات ودعم مباشر
- منصة يومية — ليس PDF أو رابط خارجي

> للتفاصيل: Master Documentation — الفصل 24 (Project Identity) و 16 (Content Strategy).

# 7. نظام الموظفين — Employee System

## 7.1 مبادئ العمل

1. **لا أحد يبدأ العمل قبل قراءة هذا الدستور.**
2. **كل مهمة لها مالك واحد** — المسؤولية لا تُقسّم ضبابياً.
3. **الشفافية** — إن لم تفهم المهمة، اسأل قبل التنفيذ.
4. **التوثيق جزء من العمل** — ليس خطوة اختيارية بعد الانتهاء.

## 7.2 مستويات الصلاحية

| المستوى | من يملكه | ماذا يستطيع |
|---------|----------|-------------|
| **L1 — تنفيذ** | موظف جديد، متعاون، AI Agent | مهام محددة النطاق ضمن ملفات مفتوحة |
| **L2 — تطوير** | مطور معتمد | بناء ميزات ضمن المرحلة، migrations جديدة، Edge Functions |
| **L3 — منتج** | قائد منتج | تحديد أولويات، قبول ميزات، فتح مناطق محمية |
| **L4 — استراتيجي** | المؤسس | قرارات المراحل، تغيير الفلسفة، توسع السوق |

## 7.3 التقييم والمساءلة

- **الجودة** تُقاس بـ: هل الميزة تعمل؟ هل موثّقة؟ هل لا تكسر الفانل؟
- **الالتزام بالمرحلة** أهم من «إنجاز سريع» — Placeholder مقبول مؤقتاً؛ ادعاء الاكتمال غير مقبول
- **الأخطاء** تُعالج بالشفافية: توثيق المشكلة → إصلاح → تحديث Master Documentation إن لزم

## 7.4 العمل مع أدوات AI (Lovable / Cursor)

- AI **ينفّذ** ضمن قواعد الدستور — لا يقرّر استراتيجياً
- لا force-push أو تعديل Git history المدفوع
- Commit فقط عند طلب صريح من المؤسس أو قائد المنتج
- أي تغيير في Landing/Quiz/Admin يتطلب موافقة L3+

# Golden Rule

أي موظف يعمل **فقط** داخل حدود اختصاصه.

إذا كان الطلب خارج مسؤوليته فلا يحاول اتخاذ القرار أو التفكير بدلاً من موظف آخر.

بل يجيب بالشكل التالي:

```
هذه المهمة ليست ضمن مسؤوليتي.

الموظف المناسب:
XXXXXXXX

سبب التحويل:
XXXXXXXX
```

**يمنع تماماً:**

- تعديل تخصص موظف آخر.
- اتخاذ قرار معماري إذا لم يكن Architect.
- اتخاذ قرار تصميم إذا لم يكن Designer.
- اتخاذ قرار منتج إذا لم يكن Product Owner.
- تنفيذ كود إذا لم يكن Developer.

# 8. قواعد اتخاذ القرار — Decision Making Rules

## 8.1 هرم القرارات

```
قرار استراتيجي (مرحلة، فلسفة، تسعير)
    → المؤسس فقط

قرار منتج (ميزة، أولوية، UX جوهري)
    → قائد المنتج + استشارة المؤسس

قرار تقني (بنية، migration، مكتبة)
    → المهندس + موافقة قائد المنتج

قرار تشغيلي (دفع، دعم عميل، إيميل)
    → الأدمن التشغيلي ضمن الإجراء المعتمد
```

## 8.2 قواعد ملزمة

| # | القاعدة |
|---|---------|
| D1 | **Master Documentation = مصدر الحقيقة التقنية** — لا افتراضات من الذاكرة |
| D2 | **الدستور = مصدر الحقيقة الإدارية** — لا استثناءات صامتة |
| D3 | عند الشك: **اسأل قبل البناء** — خاصة في المناطق المحمية |
| D4 | **لا قرار معماري دون توثيق** — إن غيّر القرار السلوك، يُحدَّث Master Doc |
| D5 | **المرحلة الحالية تحدد النطاق** — لا Phase 4 قبل إكمال Phase 2 |
| D6 | **القرارات الثابتة** (انظر §10.2) لا تُعاد مناقشتها دون سبب استراتيجي |

## 8.3 تصعيد القرار

1. الموظف يوثّق السؤال/الخيارات (2–3 بدائل كحد أقصى)
2. يرفع لمالك المنطقة (انظر §6.3)
3. إن تعارضت المناطق → قائد المنتج
4. إن تعارضت الاستراتيجية → المؤسس

# 9. قواعد التوثيق — Documentation Rules

## 9.1 هرم الوثائق

| المستوى | الوثيقة | الغرض | الجمهور |
|---------|---------|-------|---------|
| **1 — دستور** | `PROJECT_HANDBOOK.md` (هذه الوثيقة) | رؤية، قيم، قواعد، هيكل | كل موظف جديد — إلزامي |
| **2 — مرجع تقني** | `MASTER_PROJECT_DOCUMENTATION.md` | حقائق الكود، DB، مسارات، أمان | مطورون، منتج، AI Agents |
| **3 — تشغيلي** | `AGENTS.md` | قواعد Lovable/Git | مطورون |
| **4 — محلي** | `scripts/`, تعليقات الكود | تفاصيل تنفيذية | المنفّذ فقط |

## 9.2 متى تُحدَّث الوثائق

| الحدث | الوثيقة المطلوب تحديثها |
|-------|------------------------|
| تغيير استراتيجي أو قيمة | الدستور (هذه الوثيقة) |
| إضافة/حذف ميزة، route، جدول DB | Master Documentation |
| تغيير قواعد Git/Lovable | AGENTS.md |
| إضافة تمارين أو تغيير هيكل المكتبة | `exercise-library.json` + Master §13 |

## 9.3 قواعد الكتابة

- **العربية** هي لغة الوثائق الإدارية والاستراتيجية
- **الإنجليزية** مسموحة للمصطلحات التقنية والمسارات (`/app`, RPC, RLS)
- **لا تكرار**: الدستور يحيل للـ Master — لا ينسخ جداول DB أو مسارات
- **الصدق**: ما هو Placeholder يُكتب Placeholder؛ ما هو غير موجود يُكتب «غير موجود»
- **الإصدار والتاريخ** يُحدَّثان عند كل مراجعة رسمية

## 9.4 ممنوع

- وثائق متعارضة دون تحديد أيهما أعلى
- README فارغ كبديل عن التوثيق الرسمي
- قرارات شفهية دون تسجيل في الوثيقة المناسبة

# 10. قواعد التطوير — Development Rules

> التفاصيل التقنية الكاملة: Master Documentation — الفصل 3, 4, 22, 23.

## 10.1 مبادئ التطوير

| # | المبدأ |
|---|--------|
| 1 | **أقل تغيير يحقق الهدف** — لا إعادة هيكلة واسعة دون طلب |
| 2 | **لا تلمس المناطق المحمية** (Landing, Quiz, Admin) إلا بطلب صريح |
| 3 | **لا تعدّل ملفات مُولَّدة** (`routeTree.gen.ts`, `client.ts`, `.vercel/output/`) |
| 4 | **Migrations جديدة** — لا تعديل migrations منشورة |
| 5 | **اختبر الفانل** بعد أي تغيير في Quiz أو Checkout |
| 6 | **`npm run build` يجب أن ينجح** قبل اعتبار المهمة مكتملة |

## 10.2 قرارات ثابتة (Non-Negotiable)

| القرار | القيمة المعتمدة |
|--------|-----------------|
| مسار المنصة | `/app` (ليس `/dashboard`) |
| Upgrade CTA | → `/quiz` |
| Coach Chat (مؤقت) | WhatsApp — `seed-content.ts` |
| `payment_status` الإنتاجي | `approved` |
| المنتج | رقمي بالكامل — ليس حضورياً |
| Commit | فقط عند طلب صريح |
| Git history | لا force-push (Lovable) |

## 10.3 نطاق العمل حسب المرحلة

| المرحلة | مسموح | ممنوع |
|---------|-------|-------|
| **Phase 1** (حالية) | `/app` shell، seed content، gating | برنامج حقيقي، تغذية، DB محتوى |
| **Phase 2** (قادمة) | برنامج، تغذية، progress، membership RPC | admin محتوى، متجر، Paddle |
| **Phase 3+** | جداول محتوى، discover، مدفوعات متقدمة | — يُفتح بقرار استراتيجي |

## 10.4 التزامن مع Lovable

- المشروع متصل بـ Lovable — الفرع يجب أن يبقى قابلاً للعمل
- لا force-push، rebase، amend، أو squash لcommits مدفوعة
- راجع `AGENTS.md` قبل أي عملية Git غير اعتيادية

# 13. قواعد تسليم المهام — Task Handover Rules

## 13.1 قبل البدء

كل مهمة جديدة يجب أن تحتوي:

- [ ] **الهدف** — ماذا نريد تحقيقه؟
- [ ] **النطاق** — أي ملفات/مسارات؟ أي مناطق محمية؟
- [ ] **المرجع** — أي فصل في Master Doc؟
- [ ] **معايير القبول** — كيف نعرف أنها اكتملت؟
- [ ] **المالك** — من المسؤول؟
- [ ] **الموعد** — متى؟

## 13.2 عند التسليم

| العنصر | مطلوب |
|--------|-------|
| ملخص التغيير | ماذا تغيّر ولماذا |
| حالة البناء | `npm run build` ناجح |
| تحديث الوثائق | إن تغيّرت حقيقة تقنية → Master Doc |
| المخاطر المعروفة | أي شيء لم يُختبر أو يحتاج متابعة |
| الخطوة التالية | ماذا يأتي بعد هذا العمل |

## 13.3 تسليم بين موظفين / AI Agents

```
المالك السابق يُسلّم:
  1. سياق القرار (لماذا بدأنا)
  2. ما اكتمل وما لم يكتمل
  3. الملفات المتأثرة
  4. Blockers مفتوحة
  5. مرجع Master Doc المحدّث

المالك الجديد يُقرّ بـ:
  1. قراءة الدستور (إن لم يقرأ)
  2. قراءة فصول Master Doc ذات الصلة
  3. فهم معايير القبول
  4. تأكيد النطاق — لا توسيع صامت
```

## 13.4 ممنوع في التسليم

- «اتصلت به» دون توثيق
- تسليم بدون معرفة حالة البناء
- تسليم مع تغييرات في مناطق محمية دون إفصاح
- افتراض أن الطرف التالي يعرف السياق

# 14. مبادئ خارطة الطريق — Roadmap Principles

> التفاصيل: Master Documentation — الفصل 21.

## 14.1 فلسفة المراحل

```
كل مرحلة = قيمة حقيقية للعضو — لا هيكل فارغ
```

## 14.2 المراحل المعتمدة

|---------|-------|--------|

4. هل تحتاج تغييراً في الدستور أو Master Doc؟
5. من المالك وما معيار القبول؟

**إن كانت الإجابة على (2) «واجهة فارغة» — تُؤجَّل أو تُبنى كـ Placeholder صريح.**

# 15. معايير الشركة — Company Standards

## 15.1 معايير المنتج

| المعيار | التعريف |
|---------|---------|
| **صادق** | لا ميزة مُعلَنة إلا إن كانت تعمل فعلاً |
| **عربي** | RTL، لغة، صور — للجمهور العربي |
| **يومي** | المنصة تُستخدم كل يوم — ليس مرة واحدة |
| **مخصص** | الكويز يؤثر على ما يُعرض — ليس برنامجاً عاماً |
| **مدعوم** | Premium يصل للكوتش — WhatsApp الآن؛ in-app لاحقاً |

## 15.2 معايير الجودة التقنية

| المعيار | الحد الأدنى |
|---------|-------------|
| البناء | `npm run build` ينجح |
| الأمان | RLS + RPC — لا SELECT مباشر على `leads` |
| الأسرار | لا commit لـ `.env` بقيم حقيقية |
| التوثيق | Master Doc محدّث عند تغيير حقيقة تقنية |
| Git | لا force-push؛ فرع قابل للعمل على Lovable |

## 15.3 معايير المحتوى

| المعيار | التعريف |
|---------|---------|
| التمارين | من `exercise-library.json` — 320 تمرين، 13 مجموعة |
| الفيديو | `exercise.mp4` + `instructions.mp4` + `metadata.json` |
| التغذية | غير موجودة بعد — لا محتوى عشوائي يُقدَّم كخطة |
| اللغة | عربية واضحة — مصطلحات رياضية مفهومة |

## 15.4 معايير التشغيل

| المعيار | الإجراء |
|---------|---------|
| مراجعة الدفع | خلال وقت عمل معقول — عبر `/admin/payments` |
| إيميل الدعوة | يُرسل فور القبول — عبر `admin-accept-payment` |
| إشعار الأدمن | يُرسل عند رفع إيصال — `notify-receipt-upload` |
| Supabase | migrations وEdge Functions مطبّقة على الإنتاج |

## 15.5 ما نقبله وما لا نقبله

| ✅ نقبل | ❌ لا نقبل |
|---------|-----------|
| Placeholder صريح بمرحلة مخططة | ادعاء اكتمال ميزة وهمية |
| تحسينات صغيرة مركّزة | إعادة هيكلة واسعة دون طلب |
| توثيق قبل الكود (للقرارات الكبيرة) | قرارات شفهية بلا أثر |
| اختبار الفانل بعد تغيير حساس | دفع كود مكسور لـ `main` |
| مراجعة الكوتش للمحتوى التدريبي | نشر تمارين/وجبات دون مراجعة |
